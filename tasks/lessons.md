# Lessons

Project-specific lessons captured from user corrections and gotchas. Reviewed at session start. Update on any correction — write the rule behind it, apply same session.

## Format

- **[YYYY-MM-DD] Short rule** — what went wrong, what to do instead, why.

## Entries

- **[2026-07-12] rembg on Python 3.10 needs three-way pin** — `rembg[cpu]==2.0.69` is the last release supporting py3.10 (2.0.70+ requires 3.11). Bare `numpy`/`onnxruntime` resolve wrong: numpy latest needs py3.12 (pin `>=1.26,<2.3`, 2.2.x is the 3.10 ceiling); onnxruntime latest lacks cp310 macOS-arm64 wheels (pin `>=1.19,<1.20` — 1.19.2 ships cp310 for mac-arm64 dev + linux-x64 HF). Why: strict `requires-python = ">=3.10,<3.11"`. Applies: any new sci-py-adjacent dep here — dry-run `poetry add --dry-run` and check wheel abi tags before committing.
- **[2026-07-12] Plans must account for tests of code being deleted/refactored** — Plan said "delete `apply_geometry_and_colour`" and "change the router's `apply_edits` call path" but missed that `test_edit_pipeline_geometry.py` tests that function directly and `test_import_shots_api.py` monkeypatches that exact seam — both break silently. Before writing a refactor/delete task, `grep` the symbol across `tests/` and fold test migration into the task. Also: module-global caches leak across tests — any test hitting a cached path needs an autouse clear fixture. Why: green suite is an implied Global Constraint; a task that breaks sibling tests isn't done.
- **[2026-07-12] BiRefNet-lite ~8s/image warm on CPU** — `birefnet-general-lite` cutout on mac-arm64 CPU is ~8s/image (1144×811). Design for one-at-a-time interactive use with debounced preview; never batch-block a request. Full `birefnet-general` (~928MB) is heavier still. See [[dependency pin lesson]].
- **[2026-07-15] The Drive service account can read and move, but NEVER create** — a service account has no storage quota of its own. A file it creates in a My Drive folder counts against the creator's quota (zero), so every upload 403s `storageQuotaExceeded`. Permission is a red herring: the folder reports `canAddChildren: true` and it still fails. Both Google remedies (Shared Drive, OAuth delegation) require Workspace; this account is personal Gmail (`bindalscreations@gmail.com`) and the SA sees zero Shared Drives — so this is a permanent constraint, not config. Backfill works only because it *moves* human-owned files. Rule: anything that needs to persist a **generated** image goes to Supabase Storage (see `mockups-temp` staging in `batch_worker`), never Drive. Cost of learning it: a merged feature that was 100% broken in prod.
- **[2026-07-15] Mocking an integration mocks away its provider-enforced rules** — the whole batch Drive-staging path passed 282 tests, then failed on every real call. `upload_image` was only ever exercised against a fake that happily returned `{"id": ...}`; the quota rule lives in Google's API, not in our code, so no amount of unit testing could see it. Rule: for any NEW verb against an external API (not just a new caller of an existing one), hit the real service once — a throwaway probe script against one record — BEFORE building on it. Applies double to a plan's stated assumptions: this plan asserted "the SA owns files it uploaded" as fact, and that single unverified sentence invalidated ~6 tasks of work.
- **[2026-07-15] Expand/contract for schema changes — a rename against a running backend is an outage** — applied `drive_file_id -> storage_path` to prod before deploying the code that uses it; the live backend kept selecting the old columns and every batch request became "Database request failed". Rule: never rename/drop a column while old code is live. Add the new column → deploy code that writes both → backfill → drop the old one in a later migration. If the deploy path is slow (HF rebuild ~20 min, see below), that window is long and very visible. Applies to any migration on this shared Supabase project — the Inventory app reads these tables too.
- **[2026-07-15] A test that stubs the seam you changed will pass through a broken change** — added `_concurrency()` reading `settings.batch_concurrency`, forgot the `settings` import; 8 worker tests stayed green because every pool test monkeypatched `_concurrency` itself. A `NameError` would have shipped. Rule: when a function's job is to *read config*, one test must exercise the real read (env var set/unset/garbage) with nothing stubbed. Green tests are evidence only for the code paths they actually execute — check the import resolves (`python -c "import mod; mod.fn()"`) before trusting the suite.
- **[2026-07-15] Merging to main auto-deploys; frontend and backend land minutes apart** — `.github/workflows/sync-to-hf.yml` force-pushes `main` to the HF Space on every push, so merge = deploy, no approval step. Vercel serves the new frontend in seconds while the HF Docker image rebuilds (~20 min on cpu-basic; it sat in `RUNNING_APP_STARTING` and served the OLD container the whole time). Symptom: new frontend calls new routes → **404 Not Found**, which reads exactly like a routing bug. Rule: before debugging a 404 after a merge, check `curl <space>/openapi.json` for the route and the Space `sha` vs `origin/main` — if the route is absent, it's deploy skew, not code. Discriminator: a mounted-but-authed route returns 401, an absent one returns 404.
- **[2026-07-15] Docker: COPY app code LAST or every push rebuilds the world** — the Dockerfile copied `mockup_generator/`+`backend/` before `pip install .`, so any one-line code change invalidated the install layer AND the 214 MB BiRefNet model download. Fixed in `e879ec0`: install deps against a stub package from `pyproject.toml`/`poetry.lock` alone, cache the model, then copy code and `pip install --no-deps .` to swap the stub. Rule: layers ordered least→most volatile. Note poetry-core accepts a stub package (just `__init__.py`) — verified by building a wheel.
- **[2026-07-15] A column comment that describes behaviour is a claim — verify it against the code** — `2026-07-15-batch-storage-staging.sql` documented `storage_path` as "Cleared on accept/reject/edit"; the code only cleared it on edit. Accept deleted the staged object and left the path, so History signed a URL for a file that no longer existed and rendered a broken image. The migration comment read as truth and hid the gap for a whole feature's lifetime. Rule: when a doc/comment states behaviour, grep the call sites before trusting it — and when writing one, write it about what the code does, not what the plan intended.
- **[2026-07-15] Ask what the screen must SHOW before adding state to persist** — "History shows a broken image" looked like it needed a `published_url` column (migration + repo + router + tests). The user's actual requirement was a list of Product ID/Color/Status — no image at all — so the whole column was deleted minutes after being written. The correct fix was one line (null the dangling path) plus a table. Rule: for any bug whose fix introduces persisted state, confirm the display requirement first; the cheapest fix for "the image is gone" is often "this view doesn't need an image".
- **[2026-07-15] Reuse the loading pattern the app already has before inventing a slow one** — batch review inlined every full-res Drive source + the 4K mockup as base64 in one serial JSON response (tens of MB) while the rest of the portal already used thumbnail-then-lazy-large (`useImageLightbox.showDrive` + `/api/drive/image/{id}`) and signed Storage URLs. The slow path wasn't a hard problem, it was an unshared one. Rule: before writing an image/data-fetch path, grep for how the neighbouring tab does it.
- **[2026-07-15] Systemic failure in a queue: stop the bleeding before diagnosing** — batch cards generated (real Gemini spend) and *then* 403'd on staging, so each queued card was guaranteed to waste money. Flipped the 10 remaining `queued` rows to `failed` with an explanatory error before investigating; only 2 cards ever paid. Rule: when every item in a worklist will fail for the same reason, pause the queue first — it takes one guarded UPDATE, it's reversible via the existing Retry path, and diagnosis is worthless if it costs a rupee a minute to think.
- **[2026-07-15] Retry guards must be tested against the real exception object** — `generate_with_retries` checked `getattr(e, "status_code", None) == 429`, but google-genai's `APIError` stores the HTTP status as **`code`**; `status_code` is only a local in the SDK's `errors.py` and never reaches the exception. `getattr` returned `None`, so the guard silently fell through to `raise` and **every 429 failed on the first attempt with zero backoff** — 181 of 195 cards in a 100-product batch, burning a Drive download each, all inside 12 minutes (the giveaway: 40 failures/min is impossible if 8 retries with 8-60s backoff were running). The same typo sat in `prompts/refine.py`. No test ever covered the 429 path; a hand-rolled fake exception with a `status_code` attribute would have passed while prod burned. Rule: construct the **real** SDK exception in the test (`errors.ClientError(429, response_json, None)`), never a stub — and `getattr(x, "attr", None)` on a third-party object is an assumption, so grep the library's `__init__` for the attribute name before trusting it. A defensive `getattr` default turns a wrong attribute name into silent wrong behaviour instead of an `AttributeError`.
