## 4. Target architecture

**Two deployables:** a Python **backend** (FastAPI) exposing a REST API, and a **React** frontend. The existing `mockup_generator` package is refactored into a clean, framework-agnostic core that the backend imports (no Streamlit in core).

```
repo/
  mockup_generator/              # CORE — pure Python, no web framework
    __init__.py
    config.py                    # Settings: env/secrets, single source (guarded st.secrets)
    prompts/
      defaults.py                # prompt constants + CATEGORY_PROMPTS (by categoryid)
    generation/
      common.py                  # part_from_pil, load images, client factory, retry (deduped)
      images.py                  # was create_base.py
      video.py                   # was create_video.py (import fixed)
      legacy_openai.py           # was create_mockup.py (import fixed)
    integrations/
      supabase_client.py         # anon + service clients
      drive_client.py            # service-account: parse url, list, download, upload
      storage_client.py          # Supabase Storage upload + public URL
    db/
      products_repo.py           # products + category + status join
      prompts_repo.py            # prompt per categoryid
      mockups_repo.py            # read/update mockups flags
      variations_repo.py         # mockup_variations CRUD (§4.8)
  backend/                       # FastAPI app
    main.py                      # app + routers + JWT auth dependency
    routers/ (products, generate, review, prompts)
  frontend/                      # React (Vite + TS)
    src/ (auth, api client, product list, review UI)
app.py                          # LEGACY Streamlit — kept working during transition, removed later
```

**Migration note:** the refactor (Phase 0) only carves out the core package and keeps `app.py` working on top of it. React + FastAPI land in later phases. Nothing about the React decision changes the Phase-0 core extraction.

### 4.2 Data flow (target)
```
Login (Google) -> match profiles.email AND is_active -> else reject
  -> UI lists products (filterable by category; default = pending = mockups flag false)
  -> User selects product(s)
  -> For each product:
       prompt   = prompts_repo.by_category(product.categoryid)   # editable
       folderId = parse(product.producturl)
       images   = drive_client.download(folderId)                # service account
       result   = generation.images.generate(prompt, images)
       links    = drive_client.upload_outputs(result, dest)      # OUTPUT loc — see open item
       images_repo.add(productid, links)                         # productimages
       mockups_repo.set_flag(productid, base_mockup=true, ...)
  -> UI shows result + Drive link; skips products already done
```

### 4.3 Auth design
- **React** uses `supabase-js` Google OAuth → obtains a Supabase session JWT.
- **FastAPI** validates the JWT on every request (Supabase JWKS), extracts the user, looks up `profiles` by email/uid; allows only if row exists **and** `is_active=true`. `role` gates admin actions (prompt edits, activating users).
- Server-side writes (prompts, mockups, variations) use the **Supabase secret key** (service role) — *needed, add `SUPABASE_SECRET_KEY` to `.env`*.

### 4.4 Drive design (service account)
- Service-account JSON credential (*needed*). Both Google accounts share their product folders with the SA email (or move under SA-shared parents).
- `drive_client`: `parse_folder_id(url)` (3 formats), `list_images(folderId)`, `download_to_temp`, `upload_outputs`, `make_links`.
- Output destination: **OPEN ITEM** — confirm (same product folder `/mockups` subfolder, a separate output parent, or store link only in `productimages`).

### 4.5 Prompts
- `prompts/defaults.py` seeds a DB table keyed by `categoryid`; reuse the 9 existing prompts for their categories; remaining categories start empty/generic until written.
- `prompts_repo.by_category` loads for UI; editable inline; admin can persist edits.

### 4.6 Dedup / skip-already-generated
- **`mockups.base_mockup = true` indicates a final mockup already exists** → skip that product by default.
- Default product list filters to `base_mockup = false` (pending).
- `redo = true` forces regeneration even if `base_mockup` is true.
- Optionally double-check Drive for an existing output file before spending API budget.

### 4.7 Storage — where approved images live (recommendation)
**Approved images → Supabase Storage bucket** (e.g. `mockups`), path `productid/{variation}.png`. Rationale (you flagged future hosting):
- S3-compatible, **in the same Supabase project**; gives stable public/CDN URLs → React renders `<img src=…>` directly, no Drive API call or OAuth per view.
- Drive stays the **working/raw archive** (generated images already live in a Drive folder). Drive is weak for *serving* a web app (rate limits, auth, no CDN).
- **Portable:** the stored path/URL pattern moves cleanly to S3 / Cloudflare R2 / GCS later — only the bucket backend changes, not the app.

**Flow:** generate → save raw to Drive working folder → on **approve** → copy to Supabase Storage + store its URL in `mockup_variations.storage_url` (and/or `productimages.imageurl` for the published one).

### 4.8 Variations + review workflow
Multiple variations per `productid` are expected. New table **`mockup_variations`** (proposed, pending approval):

| column | type | note |
|---|---|---|
| `id` | uuid pk | |
| `productid` | varchar fk→products | |
| `variation_no` | int | 1..N per product |
| `kind` | text | base_mockup / video / etc. |
| `drive_url` | text | raw working copy |
| `storage_url` | text | set on approval (Supabase Storage) |
| `status` | text | `pending` / `approved` / `rejected` |
| `feedback` | text | reviewer's note on what's wrong |
| `prompt_used` | text | exact prompt sent |
| `created_by` / `created_at` | uuid / ts | from profile |
| `reviewed_by` / `reviewed_at` | uuid / ts | |

**Review UI (React):** side-by-side **input images vs generated output(s)**; per variation either:
- type **feedback** (what's incorrect) → saved; can trigger a re-generate with the note appended to the prompt, producing a new `variation_no`; or
- **approve** → status `approved`, copy to Supabase Storage, flip the relevant `mockups` flag (e.g. `base_mockup=true`), optionally write `productimages`.

## 5. Error handling & testing
- Keep retry/backoff in `generation/common.py`.
- Login failure / inactive user → friendly message, no app access.
- Drive access failure (folder not shared with SA) → clear per-product error, mark failed, continue batch.
- Quota/cost: optional per-user counts later (YAGNI now; status tracked via mockups).
- Tests: repos against a Supabase test schema/branch; mock `drive_client`; smoke-test generation with a fake image.
